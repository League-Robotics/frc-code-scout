# Robot-Code Generators — Spec-In, Code-Out, and the IO Seam

*The class of FRC tools that take a specification and hand you robot code: GUI skeleton
generators (RobotBuilder), vendor project generators (CTRE Phoenix Tuner X), config-driven
runtime libraries (YAGSL), and the not-yet-real AI generators. The thesis: **every one of them
optimizes time-to-driving-robot, which is the opposite axis from the rubric.** The rubric rewards
swappability, sim-testability, and vendor-decoupling — the seam in `../build-spec/elite-architecture.md`.
The generators reward minutes-to-first-drive. Two of them (CTRE's generator, YAGSL) don't just
ignore the IO seam; they actively push vendor types above it, or hide the seam inside a black box
you can't trace. The one reconciling move — generate the **constants**, own the **architecture** —
is the practical takeaway.*

> **Status:** synthesis doc from a deep-research pass (105 extracted claims across official docs +
> ChiefDelphi, adversarially verified — 74/75 verdicts confirmed high-confidence, 1 refuted and
> corrected inline). Sources in **Pointers**. This maps the generator landscape onto the seams in
> `../build-spec/elite-architecture.md` and the leak-detection lens in
> `../build-spec/code-review-principles.md`. It does **not** cover path/trajectory generators
> (PathPlanner, Choreo, the deprecated PathWeaver) — those generate *motion*, not robot code, and
> belong with the autonomy material, not here.

---

## 0. The one axis that sorts the whole landscape

Everything below answers two questions, and the answers correlate:

1. **Who owns the drivetrain/subsystem code after generation?** — you (one-shot or round-trip
   scaffold) or the tool (a black-box library you depend on).
2. **Where do the vendor SDK types live?** — below a hardware IO line you control, smeared across
   your subsystems, or hidden inside the library.

The rubric's golden rule is that `com.ctre.*` / `com.revrobotics.*` / `org.photonvision.*` must
**not appear above the IO seam** (D1), and that subsystems must be **unit-testable in sim** (D3/D4).
Measured against that, the generators land like this:

| Tool | Spec format | Round-trip model | Vendor coupling | Sim-testable OOTB | You own the code? | Rubric ceiling |
|---|---|---|---|---|---|---|
| **WPILib RobotBuilder** | GUI subsystem/command tree | Protected regions (true round-trip) | WPILib + concrete HW classes *in* subsystems; no seam | ✗ | ✓ (skeleton) | Low — **deprecated, removed 2027** |
| **CTRE Phoenix Tuner X Swerve Gen** | Wizard + characterization | One-shot (no markers) | **Maximal** — `com.ctre.phoenix6` *is* the drivetrain; all-CTRE HW required | ✓ (sim thread built in) | ✓ (own the emitted file) | High *performance*, zero seam — locked to CTRE |
| **YAGSL** (config-driven runtime lib) | JSON config directory | Black-box library (vendordep) | Hidden inside the lib (3 vendor families), version-coupled | ✓ (maple-sim) | ✗ — only the thin wrapper | Mid — can't trace/extend, AK-replay hard |
| **AI / LLM** | Natural-language prompt | N/A | Whatever it hallucinates (often deprecated APIs) | ✗ usually | ✓ | Low — no traction as of 2026 |

The rest of this doc is the evidence behind each row.

## 1. Full-project / skeleton generators — *you own the emitted code*

### 1.1 WPILib RobotBuilder — the original spec→code tool (now dying)

- **Spec:** a GUI tree. You decompose the robot into subsystems (each holding actuators, sensors,
  controllers) and commands, set properties (ports, PID gains), and bind commands to buttons.
- **Artifact:** a Java/C++/Python **skeleton** — subsystem/command declarations, button mappings,
  autonomous + default-command wiring, SmartDashboard bindings in `RobotContainer`. It emits a
  *skeleton you complete by hand*: command lifecycle methods (`initialize`/`execute`/`isFinished`/
  `end`) are stubs you fill in. It does **not** produce a finished, working subsystem.
- **Round-trip model — genuinely round-trip.** Code lives between
  `// BEGIN AUTOGENERATED CODE, SOURCE=ROBOTBUILDER ID=...` / `// END AUTOGENERATED CODE` markers.
  It regenerates *inside* the markers and preserves your edits *outside* them. Each file carries a
  `// ROBOTBUILDER TYPE: ...` header; delete the header or the markers and you **permanently
  disable** regeneration for that file (and risk losing manual additions inside protected zones on
  re-export). This is the most sophisticated round-trip model of any tool here — and it's the one
  being retired.
- **Architecture verdict — poor.** The decomposition is command-based, not IO-seam. Generated
  subsystems reference concrete hardware classes **directly** (`PWMVictorSPX`, `AnalogPotentiometer`);
  the generated `PIDSubsystem` embeds `kP/kI/kD/kF` and reads sensors / drives motors *inside* the
  subsystem `getMeasurement()`/`useOutput()`. No swappable IO layer, nothing isolated for a sim
  test. The leaking types are mostly *WPILib framework* types (`TimedRobot`, `CommandScheduler`)
  rather than `com.ctre.*`, but the structure is exactly the "subsystem owns its hardware"
  anti-pattern D1 exists to flag. The generated **command** layer is cleaner — it depends only on
  `edu.wpi.first.wpilibj2.command.Command` and talks to subsystems by method call, no vendor types.
- **The headline fact:** RobotBuilder is **officially deprecated and scheduled for removal in 2027**
  due to declining usage. The official toolchain is *retiring* the GUI-skeleton-generator approach
  outright — a strong signal about where the community thinks this idea leads.

### 1.2 CTRE Phoenix Tuner X — Swerve Project Generator (the one with real traction)

- **Spec:** a wizard. You set device IDs and characterize/verify the modules inside Tuner X
  (encoder inverts, drivetrain inverts, drivetrain verification routine), then click generate.
  Full-project generation is **FRC-only**; non-FRC users can emit just the constants file.
- **Artifact:** a complete FRC project — `TunerConstants.java` (per-module device IDs, CANcoder
  offsets, Slot0 gains, slip-current limit, CAN bus, all as **literal values from the
  characterization workflow**) plus a `CommandSwerveDrivetrain.java` subsystem and a
  `createDrivetrain()` factory. Built on **Phoenix's own proprietary Swerve API**
  (`SwerveDrivetrain`/`CommandSwerveDrivetrain`), *not* WPILib's swerve template.
- **Round-trip model — one-shot.** The header says machine-generated, but there are **no
  protected-region markers and no do-not-edit warnings**. You own the file after generation and
  edit it freely (contrast RobotBuilder's markered round-trip).
- **Architecture verdict — maximum vendor coupling, by construction.** The subsystem
  `extends TunerSwerveDrivetrain` → `com.ctre.phoenix6.swerve.SwerveDrivetrain`, parameterized on
  `TalonFX` (drive **and** steer), `CANcoder`, and `Pigeon2`; the constructors take
  `com.ctre.phoenix6.swerve.SwerveDrivetrainConstants` / `SwerveModuleConstants` and **the parent
  class constructs the hardware devices itself.** It imports `com.ctre.phoenix6.{configs,hardware,
  signals,swerve}` extensively. There is no seam — the entire drivetrain *is* the vendor SDK. It
  **requires all-CTRE Phoenix 6 hardware**: 8 Talon FX/FXS motors, CANcoder/CANdi/Talon-FXS-PWM
  steer feedback, a Pigeon 2 gyro. You cannot swap in a REV motor without abandoning the generated
  architecture.
- **The redeeming nuances — and they're real:**
  - Ships **built-in simulation** (a sim thread calling `updateSimState(...)` with battery voltage),
    so the drivetrain runs in sim out of the box.
  - Uses **FOC automatically** (Phoenix Pro + CANivore recommended-not-required; they enable
    synchronous sensor acquisition + FusedCANcoder/FusedCANdi accuracy).
  - Produces a drivable robot in **a few hours** (≈10 minutes if firmware and device IDs are
    already set) — explicitly engineered to minimize misconfiguration footguns.
  - Community consensus genuinely *favors* it for all-CTRE teams: "maintained and optimized by CTRE
    specifically for CTRE hardware; few teams can sustainably maintain a custom equivalent during
    build season." Even YAGSL **recommends CTRE's generator for all-CTRE hardware and emits a
    warning** when run on an all-CTRE config.
- **How to read this against the rubric:** the generator buys you a high *performance* ceiling and a
  low *time* floor, at the cost of the D1 seam. A team that stops at the generated output scores
  well on "does it drive" and poorly on "could you swap the vendor / replay a match / unit-test the
  module." See §4 for the move that keeps the constants and reclaims the seam.

### 1.3 REV MAXSwerve template

The REV analogue to the CTRE generator: a vendor-specific template project + REV Hardware Client
device configuration, hardcoding the REV stack (SparkMax/SparkFlex + NEO/Vortex, Through-Bore
encoder) the same way CTRE's hardcodes Phoenix 6. It surfaced in the corpus mainly as one of the
**bundled config presets inside YAGSL** (`maxSwerve`) rather than as a standalone generator with its
own deep round-trip tooling — i.e. its real life is as a config preset, not a code emitter.

## 2. Config-driven runtime libraries — *the library owns the code*

### 2.1 YAGSL (Yet Another Generic Swerve Library)

The fundamentally different model: you don't get emitted source at all — you author a **spec**, and
the library builds the drivetrain **at runtime**.

- **Spec:** a fixed **JSON directory** under `src/main/deploy/swerve/`:
  - top-level `swervedrive.json` + `controllerproperties.json`,
  - `physicalproperties.json`, `pidfproperties.json`,
  - a `modules/` folder with per-corner JSON (`frontleft/frontright/backleft/backright.json`).
  You edit JSON, never module-instantiation code. A **Flutter desktop Configurator app**
  (Windows/Linux/macOS) builds, validates, and exports the configs (with a simulator, robot
  tests/utilities, a live Gradle log view, NetworkTables + SSH connection); there's also a web
  tuning page.
- **How it runs:** a single `SwerveParser(dir).createSwerveDrive(...)` parses the directory and
  instantiates the entire `SwerveDrive` object **at runtime**. You write and own **only** a thin
  `SwerveSubsystem` wrapper and your drive commands — the drivetrain implementation lives **inside
  the library**.
- **Multi-vendor — its core advantage over CTRE's generator.** Motor controllers are picked by JSON
  `type` string across **three vendor families**: REV (SparkMax/SparkFlex with NEO/NEO 550/Vortex),
  CTRE (TalonFX incl. Falcon 500/Kraken X60, TalonSRX), **and Thrifty Nova (The Thrifty Bot)** —
  plus gyros NavX/ADIS/Pigeon and CANCoder/Through-Bore/SRX-Mag encoders. Vendor differences are
  abstracted behind an internal `SwerveMotor` wrapper.
  > **Verification correction.** The research corpus initially claimed YAGSL supports "REV + CTRE
  > only." That was **the single refuted claim** in this whole pass: YAGSL added **Thrifty Nova**
  > support in v2025.1.1, making it three vendor families. Some capabilities are still
  > vendor-gated — the `canbus` option (e.g. CANivore) is CTRE-only, and closed-loop PID runs only
  > on SparkMax + TalonFX — so vendor parity is approximate, surfaced through config.
- **Version coupling — the cost of the black box.** YAGSL is a pinned vendordep
  (e.g. `yagsl-2025.7.2.json`) and **configs break across upgrades** — fields get removed/renamed
  (`wheelDiameter`/`gearRatio`/`encoderPulsePerRotation` → `conversionFactor` + `optimalVoltage`),
  so you migrate the *JSON*, not your code. Note the reference **`YAGSL-Example` repo is now
  archived/read-only**, consolidated into the main `Yet-Another-Software-Suite/YAGSL` repo as of
  early 2026.
- **Architecture verdict — a seam you don't own.** Internally YAGSL *does* abstract vendors behind
  the `SwerveMotor` wrapper, which the author explicitly frames as "the IO layer," and it ships sim
  objects for all hardware + maple-sim support + AdvantageScope-compatible default logging. But it's
  the **library's** seam, not yours: you can't code-trace or extend the per-calculation behavior,
  which is precisely why experienced teams decline it — *"you lose control over how the swerve
  calculates,"* which hurts both teaching and on-field diagnosis. Concretely:
  - **AdvantageKit replay is famously hard on YAGSL.** The author won't support AK because AK
    requires modifying user code, conflicting with YAGSL's "self-contained library" design
    principle; bolting it on means forking YAGSL. A user who tried AK replay reported "little
    control over the swerve during replay."
  - It uses only basic `VelocityVoltage` on TalonFX — **no FOC** (unlike CTRE's generator).
  - It carries overhead from supporting many vendor permutations.
- **Where YAGSL wins:** it is explicitly designed to produce a functioning swerve drive with *little
  prior knowledge*, which makes it the right call for a team with no returning/experienced
  programmers — plus broad cross-team adoption means easy troubleshooting help. The honest framing
  from the forums: switching swerve *libraries* yields little real performance difference because
  none of them model the hard dynamics (module-rotation non-linearity, motor saturation, skew) —
  so for most teams the choice is about **ownership and learning**, not lap time.

### 2.2 The competing pattern worth noting — wrapper-at-the-controller (PurpleLib)

An alternative to both the explicit hand-written IO seam *and* YAGSL's hidden seam: libraries like
**PurpleLib** wrap at the motor-controller level and **auto-log every mechanism**, eliminating the
need to hand-write AdvantageKit-style IO classes. It's the third point on the spectrum — *automatic*
seam vs. *owned* seam vs. *hidden* seam — and a useful contrast when arguing why the build-spec
prefers the owned seam (`03-io-layer-strategy-pattern.md`).

## 3. AI / LLM generators — *no prompt-to-robot tool with traction (yet)*

The research was deliberately skeptical here, and the verdict is clear: **as of the 2026 season
there is no established prompt-to-robot-code generator with real adoption in FRC.**

- **The structural reason it fails:** FRC vendor APIs change every year and there's little training
  data (old ChiefDelphi/GitHub material dwarfs the new), so LLMs emit **outdated** code. Asked for a
  tank drive, GPT-5-mini produced a time-based/command-based hybrid using `MotorControllerGroup`
  (deprecated since 2024) and didn't actually use commands despite extending `SubsystemBase`. Tested
  prompts for "PathPlanner + AdvantageKit/CTRE Swerve + TalonFX" did **not** yield working code.
  Failure modes: poor Phoenix-6 / new-REVLib usage when no in-repo examples exist, broken
  `TalonFXSimState`, failure to follow AdvantageKit IO patterns, and unsafe behaviors (infinite
  loops, spurious init delays).
- **What teams actually use AI for** (112-voter poll): **debugging (51)** and **boilerplate
  subsystems/constants (34)** dominate; only **22** used it for complex algorithms like swerve
  kinematics; **44 reported a "No AI" policy.** Tool proximity matters — in-repo agentic tools
  (Copilot, Cursor) beat copy-pasting from a chat UI, and AI only accelerates people who already
  read and understand the output. It's enhanced autocomplete for repetitive IO classes, not an
  architecture generator.
- **The closest "FRC AI tool" isn't a code generator.** **CuratorFRC.com** is a rules / game-manual
  / event-data Q&A assistant, *explicitly not* designed to write code — and even as a RAG assistant
  it had grounding problems (failed to incorporate official Team Updates; the creator confirmed the
  model "fails to use the data it was given").
- **The two datapoints that bound the range:**
  - *Cautionary:* a vibe-coded driver-station component (Python, telling the robot which reef
    segments it could score on) **failed completely its first competition use**, had to be rewritten
    on-site because no one understood it, and the team **banned AI** afterward.
  - *Leading edge:* one experienced contributor reports **100% of their team's 2026 code was
    AI-generated and/or AI-reviewed** before merging to the competition robot's main branch — a
    small frontier of full agentic adoption existing alongside broad skepticism and explicit
    anti-"vibe-coding" pushback.
- **Implication for this project:** an LLM that *did* understand the build-spec — the IO seam, the
  inputs-struct logging contract, sim-backed tests — is exactly the gap. The generators above
  produce vendor-coupled or hidden-seam code *fast*; an architecture-aware generator is unoccupied
  territory. That's the thesis behind `scaffold-robot` / `add-subsystem` in this repo: generate the
  **seam**, not just the wiring.

## 4. The verdict against the rubric — and the one reconciling move

**Every generator here optimizes the wrong axis for the rubric.** Time-to-driving-robot and
seam-quality are close to orthogonal, and for CTRE's generator and YAGSL they're actively opposed:
the CTRE generator puts vendor types *above* any seam (the drivetrain *is* `com.ctre.phoenix6`), and
YAGSL hides the seam inside a versioned black box you can't trace. RobotBuilder's emitted subsystems
own their hardware directly and the tool is being retired. None of them, used as-shipped, lands a
team above the middle of the modularity ladder (`02-frc-37-team-survey.md`) — not because the code
is *bad* (CTRE's is fast and correct on CTRE hardware) but because the seam the rubric rewards isn't
there.

**The reconciling move** — the practical recommendation for a strong team — is to **separate the two
artifacts a generator conflates: the characterized constants and the architecture.**

- CTRE's generator's genuinely valuable output is `TunerConstants.java`: real, measured device IDs,
  CANcoder offsets, slip current, and Slot0 gains you'd otherwise hand-tune.
- **6328's AdvantageKit swerve template can ingest the Tuner X config directly** — it loads the
  generated constants but wraps them in a `ModuleIO` / `GyroIO` seam you own and can **replay**, and
  it adds a setpoint generator. (For a team already on PathPlanner + AdvantageKit IO + PhotonVision
  localization, experienced programmers explicitly recommend *this* over switching to YAGSL, since
  YAGSL's optimizations are already present in the AK template in a form you can code-trace and
  extend — e.g. grafting 254's Swerve Setpoint Generator.)

So the build-spec line is: **let a generator produce the numbers; never let it produce the
architecture.** Generate the constants, own the seam. That single rule turns every tool in this doc
from an architectural dead-end into a legitimate accelerator.
